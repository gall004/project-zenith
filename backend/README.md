# Backend Domain: Project Zenith Multi-Modal Engine

This directory contains the central brain and routing logic for Project Zenith, deployed as a **FastAPI** Python application.

## Core Responsibilities

The `/backend` domain handles all deterministic code, integrations, and ultra-low latency inference logic. It is strictly responsible for:
1. **LiveKit WebRTC Orchestration:** Establishing WebSockets and running the Pipecat agent pipelines for real-time, face-to-face visual and voice AI evaluation.
2. **REST API Contract Satisfaction:** Serving Pydantic-validated endpoints to honor webhook schemas generated by the external GCP Orchestrator (`gecx_agent`).
3. **Session Management:** Tracking and gracefully spinning down ephemeral Python memory pipelines (via native tooling like `end_vision_session`) during agent handoffs.

## Architecture

- **Framework:** FastAPI
- **Real-Time Data Backbone:** WebSockets and LiveKit Real-Time Communications
- **AI Agent Orchestration:** Pipecat 1.0.0 (Gemini Multimodal Live)
- **Validation:** Pydantic Models for strongly typed Webhook Requests & Responses

### The "Dual-Brain" Paradigm & The Great Wall
- **No Client Component execution:** The Python backend has zero awareness of React state. It treats the `frontend` only as a connected WebRTC peer that streams binary video/audio frames and listens for JSON DataChannel events.
- **Pipecat vs GECX Isolation:** As documented in `gecx-engineering.md`, the Python backend serves internal tools to the edge-deployed Pipecat agent. It MUST NEVER embed Google Cloud Dialogflow parameters or intents directly inside its codebase. The backend handles the execution; Vertex/CX handles the intent mapping.

## Development

Do not run Python dev servers directly within this directory using raw `uvicorn` unless you are isolating specific API unit tests. Leverage the monorepo root orchestration standard (`make dev` or root terminal start scripts) to ensure your backend receives the unified `.env` configurations.
